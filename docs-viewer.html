What is Algolia Arian?

Algolia Arian is a Partner Intelligence Platform that identifies displacement opportunities by finding companies using partner technologies (Commerce, CMS, MarTech) who are NOT using Algolia.

Core Features

  • Multi-Source Enrichment: SimilarWeb (traffic), BuiltWith (technology), Apollo (contacts & intent), Yahoo Finance (financials), SEC filings (investor intelligence)
  • Strategic Insights: Two-level insight system with module-level insights and company-level Algolia value prop synthesis
  • Browser Automation: Real-time search audits with Playwright + WebSocket live streaming
  • AI Copilot: Anthropic Agent SDK with RAG for self-service onboarding and context-aware assistance
  • 100% Validation: All data validated with confidence scores 8.0-10.0, evidence URLs tracked

Key Principles

  • Composite Key Architecture: All data linked via (company_id, audit_id) for point-in-time snapshots
  • Audit Versioning: Full history preserved - each audit creates NEW rows without overwriting
  • Data Freshness: 12-month hard cutoff (except 3-year financials), automatic refresh tracking
  • Two Features, One Platform: Partner Intelligence + Search Audit SaaS in single codebase

Data Flow Pattern

  1. Company Created: Base company record with attributes only (no audit data)
  2. Audit Triggered: New audit record created with type (partner-intel or search-audit)
  3. Enrichment Pipeline: 5 parallel agents fetch data from external APIs
  4. Module Insights: Each enrichment table stores insights from its data source
  5. Strategic Synthesis: Strategic Analysis Engine reads ALL modules, generates company-level analysis
  6. Browser Audit (Optional): Real-time search audit with live streaming
  7. Versioning: All data linked via (company_id, audit_id) - full history preserved

Composite Key Pattern

Audit versioning architecture

Core Principle All audit-scoped data uses composite primary key (company_id, audit_id, domain_key) for point-in-time snapshots.

Why Composite Keys?

Each audit creates a complete snapshot without overwriting previous data. This enables:

  • Full History: Track changes over time (traffic trends, tech stack evolution)
  • Audit Comparison: Compare February 2026 vs March 2026 data side-by-side
  • Data Integrity: Never lose historical intelligence
  • Version Control: Every audit is immutable once complete

Example: Costco Traffic Over Time

-- February 2026 Audit
INSERT INTO company_traffic (company_id, audit_id, month, visits, bounce_rate)
VALUES ('costco-uuid', 'feb-2026-audit', '2026-01', 150000000, 0.42);

-- March 2026 Audit (NEW ROW, doesn't overwrite)
INSERT INTO company_traffic (company_id, audit_id, month, visits, bounce_rate)
VALUES ('costco-uuid', 'mar-2026-audit', '2026-02', 155000000, 0.39);

-- Both audits preserved - can compare bounce rate improvement

Composite Key Pattern

Step 1: UNIQUE Constraint on Audits Table

CREATE TABLE audits (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  company_id UUID NOT NULL REFERENCES companies(id),
  audit_type VARCHAR(20) NOT NULL CHECK (audit_type IN ('partner-intel', 'search-audit')),
  created_at TIMESTAMPTZ DEFAULT NOW(),

  -- This constraint enables composite foreign keys
  UNIQUE (company_id, id)
);

Step 2: Composite Primary Key on Data Tables

CREATE TABLE company_traffic (
  company_id UUID NOT NULL,
  audit_id UUID NOT NULL,
  month VARCHAR(7) NOT NULL,  -- Domain key (2026-01)

  -- Data columns
  visits BIGINT,
  bounce_rate NUMERIC(4,3),
  insight TEXT,  -- Module-level insight (Migration 008)
  confidence_score NUMERIC(3,1),
  evidence_urls TEXT[],

  -- Composite primary key
  PRIMARY KEY (company_id, audit_id, month),

  -- Composite foreign key
  FOREIGN KEY (company_id, audit_id)
    REFERENCES audits(company_id, id) ON DELETE CASCADE
);

Step 3: Query Pattern

-- Get Costco's March 2026 traffic data
SELECT * FROM company_traffic
WHERE company_id = 'costco-uuid'
  AND audit_id = 'mar-2026-uuid';

-- Get latest traffic data for Costco (any audit)
SELECT * FROM company_traffic_latest
WHERE company_id = 'costco-uuid';

-- Compare February vs March audits
SELECT
  t1.month,
  t1.visits AS feb_visits,
  t2.visits AS mar_visits,
  t2.visits - t1.visits AS growth
FROM company_traffic t1
JOIN company_traffic t2 ON t1.company_id = t2.company_id AND t1.month = t2.month
WHERE t1.company_id = 'costco-uuid'
  AND t1.audit_id = 'feb-2026-uuid'
  AND t2.audit_id = 'mar-2026-uuid';

Strategic Insights Migration 008

Two-level insight system for Algolia value prop mapping

Status: Complete (March 7, 2026) Migration 008 adds strategic insight columns to 12 enrichment tables + company-level synthesis table.

Architecture Overview

Two-level insight system mirrors the Algolia Search Audit skill pattern:

Level 1: Module-Level Insights

Added insight, confidence_score, evidence_urls columns to 12 enrichment tables. Each table stores insights from ITS data source only.

Example: company_traffic has insights about bounce rate, mobile traffic, page views, etc. from SimilarWeb data.

  9.2,

Level 2: Company-Level Strategic Analysis

New table: company_strategic_analysis synthesizes ALL module insights into Algolia value prop mapping.

CREATE TABLE company_strategic_analysis (
  company_id UUID NOT NULL,
  audit_id UUID NOT NULL,

  -- Algolia Value Prop Mapping
  primary_value_prop VARCHAR(100) NOT NULL,
  secondary_value_props VARCHAR(100)[] NOT NULL DEFAULT '{}',

  -- Sales Intelligence
  sales_pitch TEXT NOT NULL,
  business_impact TEXT NOT NULL,
  strategic_recommendations TEXT NOT NULL,

  -- Timing Intelligence
  trigger_events TEXT[] NOT NULL DEFAULT '{}',
  timing_signals TEXT[] NOT NULL DEFAULT '{}',
  caution_signals TEXT[] NOT NULL DEFAULT '{}',

  -- Metadata
  overall_confidence_score NUMERIC(3,1) NOT NULL CHECK (overall_confidence_score >= 8.0),
  insights_synthesized_from TEXT[] NOT NULL,
  analysis_generated_at TIMESTAMPTZ DEFAULT NOW(),

  PRIMARY KEY (company_id, audit_id),
  FOREIGN KEY (company_id, audit_id) REFERENCES audits(company_id, id) ON DELETE CASCADE
);

Linking Pattern

All insights linked via composite key (company_id, audit_id):

-- Get Costco's March 2026 strategic analysis
SELECT * FROM company_strategic_analysis
WHERE company_id = 'costco-uuid' AND audit_id = 'march-2026-uuid';

-- Get ALL module insights for this audit
SELECT 'traffic' as module, insight, confidence_score, evidence_urls
FROM company_traffic
WHERE company_id = 'costco-uuid' AND audit_id = 'march-2026-uuid'
UNION ALL
SELECT 'financials', insight, confidence_score, evidence_urls
FROM company_financials
WHERE company_id = 'costco-uuid' AND audit_id = 'march-2026-uuid'
UNION ALL
SELECT 'hiring', insight, confidence_score, evidence_urls
FROM company_hiring
WHERE company_id = 'costco-uuid' AND audit_id = 'march-2026-uuid';

Backend Implementation

Service: backend/services/strategic-analysis-engine.ts

Main function: synthesizeCompanyAnalysis(companyId, auditId)

Process (mirrors Algolia Search Audit skill):

  1. Gather All Insights - Query ALL enrichment tables with WHERE company_id = X AND audit_id = Y
  2. Identify Primary Value Prop - Map insights to Algolia value props, calculate severity × confidence score
  3. Calculate Business Impact - Quantify business impact using financial + traffic data
  4. Generate Sales Pitch - Synthesize narrative from high-confidence insights
  5. Extract Timing Intelligence - Trigger events, timing signals, caution signals
  6. Save to Database - Insert into company_strategic_analysis table

Algolia Value Props:

  • search_relevance - Poor search results, high bounce rate
  • scale_performance - High traffic, slow search
  • mobile_experience - High mobile traffic, poor mobile search
  • conversion_optimization - Low conversion, cart abandonment
  • personalization - Generic experience, no recommendations
  • time_to_market - Legacy tech stack, slow development
  • operational_efficiency - High costs, manual processes

Browser Automation Phase 1D

Real-time search audits with live streaming

Core Component (Not Post-MVP) Without live browser testing, platform is just a data aggregator. Users MUST see real-time progress for trust/transparency.

Why Core?

Browser automation is the PRIMARY value prop, not a bolt-on feature:

  • Trust & Transparency: Users see live browser actions, not black-box results
  • Validation: Screenshots prove every finding with visual evidence
  • Differentiation: Competitors provide static reports - we show live testing
  • User Engagement: Live streaming creates "wow" moment

Event Stream Protocol

Frontend receives real-time events via Socket.IO:

// Event types
type BrowserEvent =
  | { type: 'test.start', testName: string, url: string }
  | { type: 'test.action', action: string, details: string }
  | { type: 'test.screenshot', screenshotUrl: string }
  | { type: 'test.result', testName: string, status: 'pass' | 'fail', score: number }
  | { type: 'audit.progress', current: number, total: number }
  | { type: 'audit.complete', totalScore: number }
  | { type: 'audit.error', error: string };

// Frontend usage
socket.on('browser-event', (event: BrowserEvent) => {
  switch (event.type) {
    case 'test.start':
      console.log(`Starting test: ${event.testName}`);
      break;
    case 'test.screenshot':
      displayScreenshot(event.screenshotUrl);
      break;
    // ... handle other events
  }
});

Search Audit Test Flow

  1. User triggers audit - Frontend calls POST /api/audits
  2. Job created - BullMQ queues browser audit job
  3. WebSocket connects - Frontend subscribes to audit events
  4. Browser launches - Playwright starts headless Chrome
  5. Tests execute - 20-step audit with screenshots
    • Homepage load
    • Search query "test product"
    • Results validation
    • Typo tolerance test
    • Empty query handling
    • Mobile viewport test
    • ... 14 more tests
  6. Live streaming - Each action sends WebSocket event
  7. Screenshots saved - Uploaded to Supabase Storage
  8. Results stored - Written to search_audit_tests table
  9. Audit complete - Final score + report generated

RAG Architecture

Documentation embedded in pgvector for semantic search:

-- documentation_embeddings table
CREATE TABLE documentation_embeddings (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  doc_type VARCHAR(50) NOT NULL,  -- 'guide', 'faq', 'api-reference'
  title TEXT NOT NULL,
  content TEXT NOT NULL,
  embedding vector(1536),  -- OpenAI ada-002 embeddings
  metadata JSONB,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Create vector index
CREATE INDEX idx_documentation_embeddings_vector
  ON documentation_embeddings
  USING ivfflat (embedding vector_cosine_ops);

-- Semantic search
SELECT title, content, metadata
FROM documentation_embeddings
WHERE embedding <-> $1::vector < 0.5  -- Cosine similarity threshold
ORDER BY embedding <-> $1::vector
LIMIT 5;

Enrichment Coordinator Flow

// enrichment-coordinator.ts
async function enrichCompany(companyId: string, auditId: string) {
  // Create 5 parallel jobs
  const jobs = [
    { agent: 'traffic', priority: 1 },
    { agent: 'technology', priority: 1 },
    { agent: 'people', priority: 1 },
    { agent: 'financials', priority: 1 },
    { agent: 'browser', priority: 2 }  // Lower priority (runs after enrichment)
  ];

  // Queue all jobs with BullMQ
  await Promise.all(
    jobs.map(job =>
      enrichmentQueue.add(job.agent, {
        companyId,
        auditId,
        priority: job.priority
      })
    )
  );

  // Wait for all agents to complete
  const results = await waitForCompletion(auditId);

  // Synthesize strategic analysis
  await strategicAnalysisEngine.synthesize(companyId, auditId);

  return results;
}

Views Reference

13 views for latest data access

View Pattern

All views use DISTINCT ON pattern to return most recent data per company:

CREATE OR REPLACE VIEW {table_name}_latest AS
SELECT DISTINCT ON (company_id)
  company_id,
  audit_id,
  {all_columns}
FROM {table_name}
ORDER BY company_id, audit_id DESC;

This enables simple queries for "latest" data without complex joins:

-- Get Costco's latest traffic data
SELECT * FROM company_traffic_latest
WHERE company_id = 'costco-uuid';

-- vs manual query (more complex)
SELECT * FROM company_traffic t1
WHERE t1.audit_id = (
  SELECT id FROM audits
  WHERE company_id = t1.company_id
  ORDER BY created_at DESC
  LIMIT 1
) AND t1.company_id = 'costco-uuid';

Special View: company_overview

Aggregates ALL latest data for a company into single row:

CREATE OR REPLACE VIEW company_overview AS
SELECT
  c.id AS company_id,
  c.name,
  c.domain,
  c.industry,

  -- Latest audit info
  a.id AS audit_id,
  a.audit_type,
  a.completed_at AS last_audit_at,

  -- Latest traffic (monthly totals)
  t.visits,
  t.bounce_rate,
  t.pages_per_visit,

  -- Latest financials
  f.revenue,
  f.gross_profit_margin,

  -- Technology count
  (SELECT COUNT(*) FROM company_technologies_latest WHERE company_id = c.id) AS tech_count,

  -- Competitor count
  (SELECT COUNT(*) FROM company_competitors_latest WHERE company_id = c.id) AS competitor_count,

  -- Executive count
  (SELECT COUNT(*) FROM company_executives_latest WHERE company_id = c.id) AS executive_count,

  -- Strategic analysis
  s.primary_value_prop,
  s.overall_confidence_score,
  s.trigger_events,

  -- Displacement opportunities
  (SELECT COUNT(*) FROM displacement_opportunities_latest WHERE company_id = c.id) AS displacement_count

FROM companies c
LEFT JOIN latest_audits a ON a.company_id = c.id
LEFT JOIN company_traffic_latest t ON t.company_id = c.id
LEFT JOIN company_financials_latest f ON f.company_id = c.id
LEFT JOIN latest_strategic_analysis s ON s.company_id = c.id;

Usage: Single query for dashboard company cards

SELECT * FROM company_overview
WHERE industry = 'Retail'
ORDER BY visits DESC
LIMIT 100;
