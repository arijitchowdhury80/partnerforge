What is Algolia Arian?

Algolia Arian is a comprehensive partner intelligence platform designed to identify and prioritize displacement opportunities. It helps sales teams find companies using partner technologies (Adobe AEM, Shopify, Amplience, Spryker, etc.) who are NOT using Algolia.

Core Formula

Displacement Targets = Companies Using Partner Tech − Existing Algolia Customers

Key Features

  • Multi-Source Enrichment - Integrates data from SimilarWeb, BuiltWith, Yahoo Finance, SEC Edgar, and JSearch
  • Composite Scoring - 4-factor scoring model (Fit, Intent, Value, Displacement) with 0-100 point scale
  • Partner Coverage - Tracks ~2,700 Adobe AEM targets, 32 Amplience, 28 Spryker, and more
  • Real-Time Enrichment - Background job processing with progress tracking
  • Smart Caching - 7-day TTL cache saves 25% on API costs
  • Security First - Google SSO, Row Level Security, server-side API keys

Database Design Week 1

Time-Series Architecture

Algolia Arian uses a lightweight time-series architecture where companies are minimal entities and all enrichment data is stored as point-in-time snapshots in audits.

Design Philosophy

  • Companies are lightweight entities - Just domain + name
  • Audits are point-in-time snapshots - All enrichment data in JSONB
  • Track changes over time - Query: "What was Costco's traffic 3 months ago?"

Schema Overview

Companies Table

CREATE TABLE companies (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  domain VARCHAR(255) UNIQUE NOT NULL,
  name VARCHAR(255),

  -- Metadata
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_companies_domain ON companies(domain);

Audits Table

CREATE TABLE audits (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  company_id UUID NOT NULL REFERENCES companies(id),

  -- Audit metadata
  audit_type VARCHAR(50) NOT NULL,  -- 'partner-intel' or 'search-audit'
  status VARCHAR(50) NOT NULL,      -- 'pending', 'running', 'completed', 'failed'

  -- Enrichment data (all API responses)
  data JSONB NOT NULL,

  -- Scoring
  score NUMERIC(5,2),
  score_breakdown JSONB,

  -- Metadata
  created_at TIMESTAMP DEFAULT NOW(),
  created_by UUID,
  completed_at TIMESTAMP
);

CREATE INDEX idx_audits_company_id ON audits(company_id);
CREATE INDEX idx_audits_audit_type ON audits(audit_type);
CREATE INDEX idx_audits_status ON audits(status);
CREATE INDEX idx_audits_created_at ON audits(created_at DESC);

Enrichment Cache Table

CREATE TABLE enrichment_cache (
  key VARCHAR(255) PRIMARY KEY,
  provider VARCHAR(50) NOT NULL,
  endpoint VARCHAR(255) NOT NULL,

  -- Cached data
  data JSONB NOT NULL,

  -- Cache metadata
  created_at TIMESTAMP DEFAULT NOW(),
  expires_at TIMESTAMP NOT NULL,
  hit_count INTEGER DEFAULT 0
);

CREATE INDEX idx_cache_provider ON enrichment_cache(provider);
CREATE INDEX idx_cache_expires_at ON enrichment_cache(expires_at);

Why JSONB?

  • Flexible schema - Different audit types have different data
  • Fast queries - GIN indexes for JSONB queries
  • No complex migrations - Add new fields without schema changes
  • Point-in-time snapshots - Preserve exact API responses

Example Queries

Get Latest Audit for a Company

SELECT * FROM audits
WHERE company_id = '...'
ORDER BY created_at DESC
LIMIT 1;

Get Audit History

SELECT
  created_at,
  score,
  data->'similarweb'->>'monthlyVisits' as visits
FROM audits
WHERE company_id = '...'
ORDER BY created_at DESC;

Find Hot Leads (Score > 80)

SELECT
  c.domain,
  a.score,
  a.data->'builtwith'->'technologies' as tech_stack
FROM audits a
JOIN companies c ON a.company_id = c.id
WHERE a.audit_type = 'partner-intel'
  AND a.score > 80
  AND a.created_at > NOW() - INTERVAL '30 days'
ORDER BY a.score DESC;

Performance Optimization

GIN Index for JSONB

-- GIN index for JSONB queries
CREATE INDEX idx_audits_data_gin ON audits USING GIN (data);

-- Example query using GIN index:
SELECT * FROM audits
WHERE data @> '{"similarweb": {"monthlyVisits": 100000000}}';

Partitioning (Future)

-- Partition audits by created_at (monthly)
CREATE TABLE audits_2026_03 PARTITION OF audits
  FOR VALUES FROM ('2026-03-01') TO ('2026-04-01');

Code Standards Mandatory

23 non-negotiable golden rules enforced via pre-commit hooks and CI/CD.

The Three Commandments

NON-NEGOTIABLE

  1. Every async function MUST have try-catch
  2. Every function MUST log entry/exit/errors
  3. Every function MUST have inline documentation

Code without these will be rejected in PR review.

Quality Metrics (Enforced in CI)

Metric Threshold Tool
Test Coverage ≥80% Jest/Vitest
TypeScript Strict 100% tsc --noEmit
Linting Errors 0 ESLint
Cyclomatic Complexity ≤10 per function ESLint plugin
Function Length ≤50 lines ESLint plugin
File Length ≤500 lines Manual review

The 23 Golden Rules

Error Handling (Rules 1-5)

  1. Every async function has try-catch
  2. Use custom error classes
  3. Global error handler middleware
  4. Graceful degradation for optional data
  5. Retry logic with exponential backoff

Logging & Tracing (Rules 6-8)

  1. Structured logging with Pino
  2. Log entry, exit, and errors
  3. Distributed tracing

TypeScript (Rules 9-12)

  1. TypeScript strict mode always
  2. Type everything, no implicit any
  3. Use unknown instead of any
  4. Zod for runtime validation

Testing (Rules 13-15)

  1. 80% test coverage minimum
  2. Test naming: describe(name) > it(should...when...)
  3. AAA pattern: Arrange, Act, Assert

Documentation (Rules 16-17)

  1. JSDoc for all public functions
  2. README per module

Security (Rules 18-20)

  1. Never log sensitive data
  2. Input validation with Zod
  3. Rate limiting

Performance (Rules 21-23)

  1. Cache expensive operations
  2. Parallel API calls
  3. Database query optimization

Example: Correct Error Handling

// ✅ CORRECT
async function enrichCompany(domain: string): Promise {
  logger.info({ domain }, 'Starting company enrichment');

  try {
    const traffic = await getTrafficData(domain);
    const techStack = await getTechStack(domain);

    logger.info({ domain, traffic, techStack }, 'Enrichment successful');
    return { traffic, techStack };

  } catch (error) {
    logger.error({
      domain,
      error: error.message,
      stack: error.stack,
      context: 'enrichCompany'
    }, 'Enrichment failed');

    throw new AppError(
      `Failed to enrich company: ${domain}`,
      'ENRICHMENT_ERROR',
      500,
      { domain, originalError: error.message }
    );
  }
}

Technology Stack

Frontend (Existing)

  • Framework: React 18 + TypeScript
  • Build: Vite
  • UI Library: Mantine v7
  • Styling: Tailwind CSS
  • State: React Context + TanStack Query
  • Hosting: Vercel (https://algolia-arian.vercel.app)

Backend (To Be Built)

  • Runtime: Node.js + Express
  • Language: TypeScript
  • Queue: BullMQ + Redis
  • Cache: Redis (7-day TTL)
  • Database: PostgreSQL (Supabase)
  • File Storage: S3 or Vercel Blob
  • Real-Time: Socket.IO
